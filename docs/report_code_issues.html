Code Issues Report: Traffic Officers Productivity Project

Generated: March 16, 2026

Summary: This report identifies 5 critical, 7 major, 5 moderate, and 4 minor issues. The most impactful problems are: (1) massive code duplication across analysis/ and analysis_network/, (2) duplicated data preparation logic within analysis/, (3) near-identical network construction files, (4) broken/incomplete scripts, and (5) hardcoded paths and magic numbers throughout.

Critical Issues

CRITICAL 1. Entire analysis_network/ Directory is a Near-Duplicate of analysis/

17 files in analysis_network/ are near-copies of their counterparts in analysis/. The only systematic difference is the data path prefix (analysis_network/data/ vs analysis/data/). Both directories maintain their own data/ subdirectory with parallel output files.

Files duplicated: 00_01, 00_02, 02_00, 02_00b, 02_01, 02_02, 02_03, 02_04, 02_06, 03_00, 03_01, 03_02, 03_03, 03_04, 03_99

Risk: Any bug fix or improvement must be applied in two places. Divergence between the two copies is likely and hard to detect.

Recommendation: Parameterize the analysis scripts to accept a data path or network definition as an argument. Use a single set of scripts with a configuration switch (e.g., data_dir <- "analysis/data/" set at the top or passed via command line). Delete the duplicate directory.
CRITICAL 2. 01_00_facts.R Re-implements 00_02_mk_estimation_sample.R Inline

analysis/01_00_facts.R lines 15-81 contain a near-verbatim copy of the data preparation logic from 00_02_mk_estimation_sample.R. The stated reason (line 15) is "re-running to not exclude the 0 ot officers," but this creates two divergent copies of the same processing pipeline.

Specific differences:

  • 01_00_facts.R does not filter out officers with is.na(safety_ability)
  • Both scripts independently read the raw anonymized data to add bereavement/FMLA flags
  • Both independently construct holiday indicators using almanac
Recommendation: Add a parameter to 00_02 to optionally keep all officers. Or create a separate "full sample" .rds. Then have 01_00 read the saved data file instead of re-running all the logic.
CRITICAL 3. Network Construction Files are 95%+ Identical

mkdata/01_06_mk_network.R, mkdata/01_07_mk_network_30.R, and mkdata/01_08_mk_network_180.R are nearly identical — the only differences are:

  • The input CSV filename (01_03_pre_network_90.csv vs _30.csv vs _180.csv)
  • The output RDS filename
  • 01_06 additionally computes wheel_nothave_degree (absent from 01_07, 01_08)
  • 01_07 and 01_08 omit the to_replace naming bugfix present in 01_06

Each file is ~240 lines of complex matrix operations and circular statistics functions copied verbatim. The helper functions (circ_mean, circ_mean2, circ_median2, circ_median2_exc) are defined identically in all three files.

Recommendation: Extract into a single parameterized script or function library. Pass the rolling window size as an argument. The shared helper functions should be in a utility file.
CRITICAL 4. Scripts Contain --- Section Separators That Would Cause Parse Errors

mkdata/01_04_mk_names.R (lines 87, 96, 104, 113) and mkdata/01_04b_mk_names_nodups.R (lines 82, 91, 99, 108) contain bare --- lines that are not valid R syntax. These scripts cannot be sourced or run as-is.

The code following the first --- also references variables (annual_tot_comp, totaling_group) that are not defined in the preceding code, suggesting these are disconnected code fragments pasted together.

Recommendation: Remove the broken sections or convert to properly delimited code blocks. Consider rewriting as a clean, single-purpose script.
CRITICAL 5. 01_04_mk_names.R Overwrites Output Without Warning

Line 84-85 writes two different datasets to the same file mkdata/data/01_04_mk_check.csv in sequence:

write.csv(pay_semi_agg[employee_name=='EMPLOYEE 432'],"mkdata/data/01_04_mk_check.csv")
write.csv(pay_recs[employee_name=='EMPLOYEE 373'],"mkdata/data/01_04_mk_check.csv")

The second write silently overwrites the first. This is almost certainly a bug.

Recommendation: Use distinct filenames for each output, or combine into one write.

Major Issues

MAJOR 6. Inconsistent wheel_nothave_degree Computation Across Network Windows

01_06_mk_network.R (90-day) computes wheel_nothave_degree and l_wheel_nothave_degree (lines 183-225), which measure connectedness weighted by distance from NOT having the wheel. This variable is used in the estimation sample (00_02) as demanders_interacted.

However, 01_07_mk_network_30.R and 01_08_mk_network_180.R do NOT compute this variable. If the 30-day or 180-day panels are used for analysis, l_wheel_nothave_degree will be missing.

Recommendation: Either add the computation to the 30/180-day scripts, or document that only the 90-day panel supports the full estimation specification. Best solution: unify into one parameterized script.
MAJOR 7. Raw Data Read Directly in Analysis Scripts

Multiple analysis scripts bypass the processed data pipeline and read the raw anonymized file directly:

  • analysis/00_02_mk_estimation_sample.R line 52: reads mkdata/20170803_payworkers_comp/anonymized_data_073117.txt
  • analysis/01_00_facts.R line 64: same
  • analysis/04_00_elasticity_did.R line 7: same
  • mkdata/01_04b_mk_names_nodups.R line 33: same

This creates fragile dependencies on the raw data location and bypasses the data-cleaning pipeline (which processes the same file through Stata in 01_01_mk_working.do).

Recommendation: Extract the bereavement/FMLA flag construction into mkdata/ as a proper data preparation step. Analysis files should only read processed intermediate files.
MAJOR 8. 01_06b_mk_network_volunteers.R Has No Final Save

This 79-line script reads 01_06_panel_working.rds, runs k-means twice to identify volunteer OT participants, and reconstructs the wheel for volunteers only. But it never saves any output. The final lines just print a cross-tabulation. The all_pairs object with the new columns is lost.

Recommendation: Either save the output (volunteer flags and reconstructed wheel) to an RDS file, or move this into an analysis script if it's purely diagnostic.
MAJOR 9. Google Maps API Key Handling

mkdata/00_01_mk_collisions.R line 7 has key<- commented out, and the script uses key on line 39 without defining it. Running this script requires manually uncommenting and entering an API key. The key is not loaded from an environment variable or config file.

Recommendation: Use Sys.getenv("GOOGLE_MAPS_API_KEY") or a .Renviron file. Never hardcode or comment-toggle API keys.
MAJOR 10. 03_99_compare_sims.R References a File Produced by an Old/Archived Script

Line 60 reads analysis/data/03_00_sim_frontier_cluster.rds, but no active script in analysis/ produces this file. The file analysis/old/03_00_sim_frontier_cluster.R exists in the archived directory but is not part of the current pipeline. If this file is deleted or becomes stale, the main comparison script will fail.

Recommendation: Move the frontier simulation script out of old/ back into the active pipeline, or replace the dependency with a currently active computation.
MAJOR 11. Slow O(n²) Loop in Collision Distance Computation

mkdata/00_01_mk_collisions.R lines 86-100 use nested for loops to compute the minimum Haversine distance from each collision to all special events on that date. With potentially thousands of collisions and events, this is extremely slow.

for (i in unique(collide$id_this)){
  ...
  for (j in unique(uniq_coords[analysis_workdate==this_date,]$id_that)){
    check_dist<-distHaversine(coords, c(...))
    max_now<- min(max_now, check_dist)
  }
  collide_list[i]<-max_now
}

Additionally, collide_list is pre-allocated with length nrow(uniq_coords) but indexed by collide$id_this, which is a different object. This works only because id_this is 1:N sequential.

Recommendation: Vectorize using sf::st_distance() or geodist::geodist() with date-grouped joins.

Moderate Issues

MODERATE 12. No Package Management or Dependency Declaration

There is no renv.lock, DESCRIPTION file, or Makefile declaring package dependencies. The project uses ~30 R packages and Stata SSC packages (unique, rangestat). Reproducing the exact environment is impossible without reading every script.

Recommendation: Add an renv lockfile for R package versions. For Stata, document required SSC packages. Consider a Makefile or targets pipeline to declare the full DAG.
MODERATE 13. Hardcoded Assertion Values Will Break If Data Changes

Multiple scripts contain hardcoded values in assertions that assume specific data properties:

  • analysis/00_01 line 63: stopifnot(length(all_not_id)==33)
  • analysis/01_00 line 100: stopifnot(nrow(all_pairs)==274908)
  • analysis/01_00 line 116: stopifnot(nrow(byofficer)==535)
  • mkdata/01_03 line 137: assert r(sum)==36 (36 employees with no work)
  • mkdata/01_02 line 323: assert r(sum)==44

These are useful as sanity checks for the current dataset but will break reproducibility if any upstream data changes.

Recommendation: Log the values as warnings rather than hard failures, or parameterize them with tolerances.
MODERATE 14. Missing Package Loads / Implicit Dependencies

Several scripts use functions without loading the necessary package:

  • mkdata/00_01_mk_collisions.R: Uses fread, mdy, str_replace, mp_geocode, distHaversine without any library() calls. Relies on packages being loaded in the global environment from a prior script.
  • analysis/00_04_example.R: Uses geocode, st_as_sf, st_crs, st_transform, ggplot, ggsave — only loads osmdata and data.table. References undefined variable shape.
  • mkdata/01_05_mk_map.R: Uses read_sf, geocode, st_as_sf, etc. without loading sf, tidygeocoder, ggplot2.
Recommendation: Every script should be self-contained with its own library() calls. Do not rely on the global R environment.
MODERATE 15. 00_04_example.R References Undefined Variable shape

Line 25 references st_crs(shape) and line 27 references shape, but shape is never defined in this file. It appears to be a partial copy of mkdata/01_05_mk_map.R where shape is loaded from a shapefile. This script will fail if run independently.

Recommendation: Add the shapefile read line from 01_05_mk_map.R, or merge this into that file.
MODERATE 16. analysis/01_07_mk_network_30.R Overwrites Exposure Columns In Place

In 01_07 (line 156-157) and 01_08 (line 156-157), the fold_mult result overwrites the original exposure columns:

to_replace<-colnames(all_pairs)[grep("^[0-9]+$",colnames(all_pairs))]
all_pairs[, (to_replace):=fold_mult(.SD), ...]

In contrast, 01_06 (the 90-day version) uses helper_-prefixed column names to avoid overwriting the originals (line 155-156). This means the 30/180-day files lose the raw exposure data after the wheel-weighting step, which could be a subtle bug if downstream code expects unweighted exposure columns.

Minor Issues

MAJOR 17. analysis_network/03_01_auction_sim.R Saves Outputs to Wrong Directory

The auction simulation in analysis_network/ saves some outputs to analysis/data/ instead of analysis_network/data/:

  • Markdown auction results (03_01_sim_auction_dev*.rds) are saved to analysis/data/
  • Straight auction results (03_01_sim_auction_straight*.rds) are saved to analysis_network/data/

Similarly, analysis_network/03_99_compare_sims.R reads from both analysis/data/ and analysis_network/data/, mixing results from the two parallel pipelines. This means the comparison script may be comparing results from different model specifications.

Recommendation: Audit all output paths in analysis_network/ to ensure they use a consistent data directory. This is another argument for merging the two directories.

  • Mixed use of _ and no separator in date-stamped directories (e.g., 20190811_weather vs 20251108_llm_attempt)
MINOR 19. Simulation Iteration Counts Are Inconsistent

Different simulation scripts use different iteration counts without clear justification:

  • 03_00_sim_random.R: 2500 iterations
  • 03_01_auction_sim.R: 2500 iterations
  • 03_02_sim_informal.R: 200 iterations
  • 03_03_sim_informal_reverse.R: likely similar to 03_02

The informal trade simulations run with 12.5x fewer iterations, which could affect precision of the comparisons in 03_99.

MINOR 20. View() Calls Left in Scripts

mkdata/01_04_mk_names.R lines 105, 106, 110, 111 and mkdata/01_04b_mk_names_nodups.R lines 100, 101, 106 contain View() calls. These are interactive-only functions that will fail in batch/script mode and serve no pipeline purpose.

MINOR 21. 05_00_explore.R Contains install.packages()

Line 4 calls install.packages("gender") directly in the script. This should never appear in a reproducible analysis script — it modifies the system state and may fail in restricted environments or overwrite working package versions.
